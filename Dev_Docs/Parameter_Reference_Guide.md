# Quantitative Parameters & Veto Gates Reference Guide

This document provides a detailed reference for all quantitative configuration parameters in the Standalone Backtester Application and the `Auto-Ghost` execution engine. It explains their roles, default values, mathematical behaviors, and direct impacts on trade execution.

---

## 1. Sequential Filter Pipeline Overview

Trade signals generated by the core **OTEO** indicator must pass through a two-stage filter pipeline before execution.
- **Stage 1 (Pre-Signal Gates)**: Evaluates the global market context (volatility, trend, time, manipulation state). If any Stage 1 filter triggers a veto, the signal is immediately discarded.
- **Stage 2 (Per-Expiry Gates)**: Evaluates specific cell metrics (historical pocket performance, Bayesian win rate probability, expected utility sizing) for each target expiry.

## 2. Parameter Reference Directory

### 🟢 2.1 Kalman Filter (Price Smoother)
The Kalman Filter smooths incoming tick prices to remove high-frequency microstructure noise (such as bid-ask bounce and random jitter) before feeding them to other indicators.

| UI Parameter Label | Config JSON Key | Data Type | Default Value | Recommended Range |
| :--- | :--- | :--- | :--- | :--- |
| **Enable Kalman Smoothing** | `kalman.enabled` | Boolean | `false` | `true` (standard) / `false` |
| **Process Noise Q** | `kalman.q` | Float | `1.00e-9` | `1.00e-11` to `1.00e-8` |
| **Measurement Noise R** | `kalman.r` | Float | `1.00e-7` | `1.00e-8` to `1.00e-5` |
| **Feed Smoothed Prices to Hurst** | `kalman.upstream_of_hurst` | Boolean | `false` | `false` |

#### Influence on Trade Execution
*   **Smoothing vs. Lag**: A lower $Q$ or a higher $R$ increases the smoothness of the price line but introduces phase lag. Lagging prices can cause indicators to trigger entries late, missing the pivot window.
*   **Upstream of Hurst**: Enabling `upstream_of_hurst` feeds the smoothed series into the Hurst calculator. Because Kalman smoothing removes high-frequency fluctuations, it artificially biases the Hurst exponent towards random-walk ($H \approx 0.5$) or trending ($H > 0.5$) classifications, neutralizing the mean-reverting micro-signals. **It is highly recommended to keep this disabled (`false`)** so Hurst is calculated on raw microstructure ticks.

### 📊 2.2 Hurst Exponent Veto Gate
The Hurst Exponent ($H$) measures the long-term memory of the price series. It classifies the market structure into mean-reverting, random-walk, or trending regimes.

| UI Parameter Label | Config JSON Key | Data Type | Default Value | Recommended Range |
| :--- | :--- | :--- | :--- | :--- |
| **Enable Hurst Veto** | `hurst.veto_enabled` | Boolean | `false` | `true` (safety) / `false` |
| **Rolling Window Size** | `hurst.window_size` | Integer | `300` | `150` to `500` ticks |
| **Mean-Reverting Threshold (< H)** | `hurst.mean_revert_limit` | Float | `0.44` | `0.38` to `0.45` |
| **Trending Threshold (> H)** | `hurst.trend_limit` | Float | `0.58` | `0.55` to `0.65` |
| **Allowed Regimes (Whitelist)** | `hurst.allowed_regimes` | List[str] | All three | `["mean_reverting"]` |
| **Veto if Raw H >= Threshold** | `hurst.filter_threshold` | Float/None | `None` | `0.46` to `0.52` |
| **Hurst Min Scale Cutoff** | `hurst.min_scale_cutoff` | Integer | `12` | `8` to `16` ticks |

#### Influence on Trade Execution
*   **Trade Volume vs. Trend Protection**: Setting `allowed_regimes` strictly to `["mean_reverting"]` blocks trades when the market is trending. This significantly reduces trade count (often by 50-70%) but protects the account from entering counter-trend positions during explosive breakouts.
*   **Rolling Window Size**: A window size that is too small (<100) causes $H$ to fluctuate wildly due to local noise. A window size that is too large (>600) makes the gator slow to respond, causing it to permit entries during the early stages of a fresh trend.
*   **Min Scale Cutoff**: Determines the smallest scale chunk used in the Rescaled Range (R/S) regression. Setting this too low (<8) allows bid-ask bounce noise to dominate the regression line, distorting the true Hurst value.

### 🌀 2.3 Ornstein-Uhlenbeck (OU) Veto Gate
The OU process models mean-reverting price actions. The veto checks whether the current price dynamic has a negative drift coefficient (mean-reverting) or a positive drift coefficient (explosive/trending).

| UI Parameter Label | Config JSON Key | Data Type | Default Value | Recommended Range |
| :--- | :--- | :--- | :--- | :--- |
| **Enable OU Veto** | `ou.veto_enabled` | Boolean | `false` | `true` (safety) / `false` |
| **OU Estimator Mode** | `ou.mode` | Select | `"kalman"` | `"kalman"` (dynamic) / `"ols"` |
| **OU OLS Window Size** | `ou.window_size` | Integer | `300` | `150` to `500` ticks |
| **OU Kalman Q Beta** | `ou.q_beta` | Float | `1.00e-6` | `1.00e-7` to `1.00e-5` |
| **OU Kalman R** | `ou.r` | Float | `1.00e-8` | `1.00e-9` to `1.00e-7` |

#### Influence on Trade Execution
*   **Explosive Regime Protection**: If the estimated beta parameter is positive ($\beta \ge 0$), the price behaves like an explosive process (e.g., a breakout or momentum run). The OU gate triggers a veto, preventing execution.
*   **Kalman vs. OLS**: `"ols"` mode fits parameters over a rolling window. It is subject to boundary lag. `"kalman"` mode uses recursive estimation to update parameters tick-by-tick, providing rapid detection of momentum spikes.

### ⏳ 2.4 Expiry Settings
Determines the duration of the trade contract. The system supports static sweeps (evaluating fixed contract lengths) or volatility-adaptive contract lengths.

| UI Parameter Label | Config JSON Key | Data Type | Default Value | Recommended Range |
| :--- | :--- | :--- | :--- | :--- |
| **Expiry Mode** | `expiry.mode` | Select | `"static"` | `"static"` or `"adaptive"` |
| **Static Expiries (seconds)** | `expiry.static_seconds` | List[int] | `[60, 120, 300]` | Expiries between `30` and `300` |
| **Adaptive Scaling C** | `expiry.adaptive_c` | Float | `10.0` | `5.0` to `20.0` |
| **Adaptive Bound Options** | `expiry.adaptive_bounds` | List[int] | `[30, 60, 120, 300]`| Allowed broker contract values |

#### Influence on Trade Execution
*   **Static Sweeps**: Evaluates signals across all listed expiries. While thorough, it trades multiple cells simultaneously.
*   **Adaptive Expiries**: Uses the formula:
    $$\text{Expiry} = C \cdot \frac{1 - H}{\text{Volatility}}$$
    Higher volatility ($V$) or lower mean reversion speed (closer to $H=0.5$) results in shorter contracts. Stronger mean reversion ($H \to 0$) or lower volatility results in longer contracts. This optimizes the contract duration to ensure the price has time to revert but does not stay in the trade long enough for a new trend to start.

### 🔮 2.5 Bayesian Credible Gate & Utility Sizer
Applies probability theory to historical win/loss ratios for specific Volatility-Liquidity-Manipulation pockets, vetoing trades with insufficient sample size or negative expected utility.

| UI Parameter Label | Config JSON Key | Data Type | Default Value | Recommended Range |
| :--- | :--- | :--- | :--- | :--- |
| **Enable Bayesian Utility** | `bayesian.enabled` | Boolean | `false` | `true` (live trading) / `false` |
| **Confidence Threshold** | `bayesian.confidence_threshold`| Float | `0.90` | `0.80` to `0.95` (80%-95%) |
| **Breakeven win rate** | `bayesian.breakeven_win_rate` | Float | `0.5208` | `0.5208` (for 92% payout) |
| **Risk Aversion Level (gamma)** | `bayesian.risk_aversion` | Float | `2.0` | `1.5` to `3.0` |
| **Max Kelly Sizing Fraction**| `bayesian.max_fraction` | Float | `0.10` | `0.02` to `0.10` (2%-10%) |

#### Influence on Trade Execution
*   **Vetoing Unproven Cells**: If a pocket has 3 wins and 0 losses, the raw win rate is 100%, but the sample size is too small. The Bayesian gate uses a Beta prior ($\alpha_0=2$, $\beta_0=2$). It calculates the probability that the true win rate exceeds the breakeven rate. For small sample sizes, this probability fails to meet the **Confidence Threshold** (e.g. 90%), triggering a veto. This prevents over-trading on random win streaks.
*   **Expected Utility Sizing**: Calculates the optimal capital allocation using Power Utility. If the expected utility calculation suggests a fraction $f \le 0.005$ (due to low win probability or high volatility), the trade is vetoed.

### 🕰️ 2.6 Timeframe Exclusion Blocks
Excludes trading during specific blocks of the day to avoid spread widening, low liquidity, or rollover anomalies.

| UI Parameter Label | Config JSON Key | Data Type | Default Value | Recommended Range |
| :--- | :--- | :--- | :--- | :--- |
| **Enable Timeframe Veto** | `timeframe.veto_enabled` | Boolean | `false` | `true` (safety) / `false` |
| **Excluded Blocks (0-5)** | `timeframe.exclusion_blocks` | List[int] | `[]` | `[0, 5]` (Rollover and Illiquid periods) |

#### Influence on Trade Execution
*   **UTC Blocks**: Day is split into 4-hour blocks:
    *   `Block 0`: 00:00 - 04:00 UTC
    *   `Block 1`: 04:00 - 08:00 UTC
    *   `Block 2`: 08:00 - 12:00 UTC
    *   `Block 3`: 12:00 - 16:00 UTC
    *   `Block 4`: 16:00 - 20:00 UTC
    *   `Block 5`: 20:00 - 24:00 UTC
*   Excluding Blocks `0` and `5` protects the engine from highly manipulated OTC market closures and spread widening during rollover hours (20:00 - 04:00 UTC).

### 🕳️ 2.7 Spike Pocket Exclusions
Allows blacklisting specific market state combinations (Volatility, Liquidity, Manipulation) and assets.

| UI Parameter Label | Config JSON Key | Data Type | Default Value | Recommended Range |
| :--- | :--- | :--- | :--- | :--- |
| **Enable Pocket Veto** | `pocket.veto_enabled` | Boolean | `false` | `true` (if using blacklist) |
| **Exclusion Cells** | `pocket.exclusion_list` | List[str] | `[]` | E.g. `Vol:LOW \| Liq:HIGH \| Manip:LOW\|60` |
| **Asset Blacklist** | `pocket.blacklist_assets` | List[str] | `[]` | E.g. `EURUSD_otc` |

#### Influence on Trade Execution
*   **Granular Filtering**: Vetoes specific cell structures. If backtests reveal that `Vol:LOW | Liq:HIGH` at a `60` second expiry performs poorly, this specific state can be blacklisted without turning off 60-second expiries globally.

### 🚨 2.8 Manipulation Gate
Scans ticks for anomalies such as pinning, artificial liquidity walls, or extreme price gaps.

| UI Parameter Label | Config JSON Key | Data Type | Default Value | Recommended Range |
| :--- | :--- | :--- | :--- | :--- |
| **Enable Manipulation Veto** | `manipulation.veto_enabled` | Boolean | `false` | `true` (safety) / `false` |
| **Severity Threshold Veto** | `manipulation.severity_threshold`| Float | `0.3` | `0.2` to `0.5` |

#### Influence on Trade Execution
*   **Veto High-Risk Events**: A threshold of `0.3` blocks signals when institutional manipulation is suspected. Higher thresholds (e.g. `0.7`) allow more trades but expose the system to flash-crash and pinning losses.

Categorizes the market structure using classical technical indicators.

| UI Parameter Label | Config JSON Key | Data Type | Default Value | Recommended Range |
| :--- | :--- | :--- | :--- | :--- |
| **Enable Regime Gate** | `regime.enabled` | Boolean | `false` | `true` / `false` |

#### Influence on Trade Execution
*   Regimes: `RANGE_BOUND`, `TREND_PULLBACK`, `TREND_REVERSAL`, `STRONG_MOMENTUM`, `BREAKOUT`, `CHOPPY`.
*   Restricting trading to `RANGE_BOUND` and `TREND_PULLBACK` ensures that the OTEO mean reversion signals are not executed during explosive breakout momentum phases.

### 🎯 2.10 OTEO Signal Gate
Sets strict constraints on the raw OTEO signal metrics.

| UI Parameter Label | Config JSON Key | Data Type | Default Value | Recommended Range |
| :--- | :--- | :--- | :--- | :--- |
| **Enable Min Z-Score Veto**| `oteo_gate.min_zscore_enabled`| Boolean | `false` | `true` / `false` |
| **Min Z-Score** | `oteo_gate.min_zscore` | Float | `0.0` | `0.5` to `1.5` |
| **Enable Max Z-Score Veto**| `oteo_gate.max_zscore_enabled`| Boolean | `false` | `true` / `false` |
| **Max Z-Score** | `oteo_gate.max_zscore` | Float | `10.0` | `3.0` to `5.0` |
| **Enable Min Score Veto** | `oteo_gate.min_score_enabled` | Boolean | `false` | `true` / `false` |
| **Min OTEO Score** | `oteo_gate.min_score` | Float | `0.0` | `20.0` to `40.0` |

#### Influence on Trade Execution
*   **Min Z-Score**: Ensures that the price movement triggering the signal represents a significant deviation (at least $X$ standard deviations).
*   **Max Z-Score**: Blocks signals generated by extreme, off-chart price jumps (e.g. news events). These spikes are often unstable and are prone to continuation rather than reversion.

## 3. Parameter Calibration Profiles

Here are three pre-configured calibration templates for different market conditions:

### 🟢 Profile A: Strict Mean Reversion (Deep Pivot Safety)
Designed to minimize drawdowns by restricting entries to highly-correlated mean-reverting states.

*   **Kalman Smoother**: `Enabled` ($Q = 10^{-9}, R = 10^{-7}, \text{upstream\_of\_hurst} = \text{false}$)
*   **Hurst Exponent**: `Veto Enabled`, `window_size` = `300`, `mean_revert_limit` = `0.42`, `allowed_regimes` = `["mean_reverting"]`
*   **OU Veto**: `Enabled` (Kalman mode)
*   **Bayesian Gate**: `Enabled` (Confidence = `0.90`, Breakeven = `0.5208`, Risk Aversion = `2.0`)
*   **Manipulation Gate**: `Enabled` (Severity = `0.30`)
*   **OTEO Gate**: `Min Z-Score` = `1.0` (Enabled), `Max Z-Score` = `4.0` (Enabled)

---

### 🟡 Profile B: Balanced Hybrid (Dynamic Adaptation)
Balances trade frequency and risk management, dynamically adapting contract durations to current volatility levels.

*   **Kalman Smoother**: `Enabled` ($Q = 10^{-9}, R = 10^{-7}, \text{upstream\_of\_hurst} = \text{false}$)
*   **Hurst Exponent**: `Veto Enabled`, `window_size` = `300`, `mean_revert_limit` = `0.44`, `allowed_regimes` = `["mean_reverting", "random_walk"]`
*   **OU Veto**: `Enabled` (Kalman mode)
*   **Expiry Mode**: `Adaptive` ($C = 10.0, \text{bounds} = [30, 60, 120, 300]$)
*   **Bayesian Gate**: `Enabled` (Confidence = `0.80`, Breakeven = `0.5208`, Risk Aversion = `1.5`)
*   **Manipulation Gate**: `Enabled` (Severity = `0.40`)

---

### 🔴 Profile C: Raw Indicator Baseline (Discovery Mode)
Bypasses veto gates to establish a raw performance baseline. Used to verify the edge of the underlying indicator.

*   **Kalman Smoother**: `Disabled`
*   **Hurst Exponent**: `Veto Disabled`
*   **OU Veto**: `Veto Disabled`
*   **Expiry Mode**: `Static` (`[60, 120, 300]`)
*   **Bayesian Gate**: `Disabled`
*   **Manipulation Gate**: `Disabled`
*   **Regime Gate**: `Disabled`
